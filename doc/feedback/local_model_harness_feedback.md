**Where the sharpest items cluster.** Two themes, and both are gaps between what the doc describes and what the runtime does.

**The harness's contract with the PM is thinner than the doc leads a PM to expect** — H1, H4, H5. The PM is told to gate a diff it never receives (H1), warned not to trust an exit code it never actually sees (H4), and invited to think of a harness specialist as accumulating context that is in fact destroyed after every dispatch (H5). Individually each is a small fix. Together they suggest the C-2 documentation and the C-2 runner were written against slightly different pictures of what a harness agent *is*. Reconciling those two pictures is the single most valuable thing this round has to offer.

**And the doc cannot stand up the agent it governs** — H6, with M5 as its security shadow. **Our expectation is that `PM_LOCAL_BACKENDS.md` *is* the rulebook for creating a C-1 or C-2 agent.** Today it is an excellent routing policy that never says a local agent is a resident process, never names the twenty environment variables that configure it, and — most seriously — never shows how a bounded schema reaches the model, so a PM who follows it exactly violates its own mandatory rule 2 and is never told. Underneath sits a structural cause: the doc is loaded *only once a local agent already exists* (`:10`, `PM.md:533`), so the one document about local agents is never read by anyone deciding to create one. Fix the trigger, then fix the content. We found all of this by trying: our operator asked whether the doc sufficed to create a C-1 and a C-2 agent, and it suffices for neither.

## What worked — please keep / reinforce

- **The two-tier C-1 / C-2 split introduced by v4.32.** Before it, the capability table read `Coding = ❌` flatly while the harness section said a 31B drives agentic coding. The reframing of that `❌` as *"a **~12B** limit, not a local-model limit"* (`PM_LOCAL_BACKENDS.md:21–22`) resolved a contradiction that would otherwise have made the doc unusable as a routing authority. **Reinforcing.**
- **"Architecture predicts reliability better than size"** (`:91`). This is the single most valuable sentence in the doc. Our own LM Studio instance offers `qwen3.6-14b-a3b` — a sparse MoE whose id advertises `14b` while activating ~3B. Without that line a PM would plausibly route coding to it. **Additive-preservation.**
- **The silent-wrong caveat** — *"sparser-MoE and smaller models fail the composite task silently (confident and wrong)"* + *"a clean `exit 0` from the CLI is not correctness"* (`:91–93`). Naming the failure mode as *silent* is what makes rule 4 feel mandatory rather than bureaucratic.
- **"`json_schema` guarantees *format*, never *content* — a bounded schema is a safety control, not a nicety"** (`:12–13`). Precise, quotable, and it pre-empts the exact mistake a reader is about to make.
- **Routing by output SHAPE, not by topic** (`:20`). This is the insight that makes the envelope teachable. Topic-based routing ("send it HA questions") would have failed immediately.
- **Real security engineering in `harness-runner.js`, not security theater.** The per-spawn client token (`:111`), the real API key never reaching the CLI (`:117–118` inject on forward; the CLI receives only the client token, `:171`/`:215`), the resolved-IP pinning against DNS rebinding (`:106`, `:120`), the unconditional 169.254/16 + IPv6-link-local metadata deny that `HOST_ALLOW` **cannot** re-open (`:108`), untrusted payload over stdin and never argv (`:282`), and the child-env allowlist with `--selftest-scrub` to prove it (`:138–149`, `:350–359`). This is materially better than what most local-model integrations ship.
- **The lazy-load framing** — *"a fleet without one pays no context for it"* (`:7–8`), enforced by the single conditional pointer at `PM.md:533`. The instinct is exactly right and we want it preserved: **keep the laziness, fix the predicate.** H6 argues the load *condition* is wrong (it fires only once a local agent exists, so it never reaches the PM deciding to create one), not that the doc should be eagerly loaded. Those are different objections and we are only making the first.
- **`backend` / `harness` / `launcher` declared mutually exclusive** (`:54`). This turned out to answer a safety question structurally rather than by judgment — see "What not to change."
- **The runner header comments are a genuinely good spec.** `local-runner.js:1–40` and `harness-runner.js:1–50` state the resident-poll requirement, the C-0 mode, the full env contract, and the H1/H2 security rationale — including the `AUTONOMY (audit CRITICAL)` note explaining *why* a one-shot runner would be wrong. Idempotent self-registration so the launcher-less path works (`local-runner.js:221–224`) is a thoughtful touch. Our H6 is not that this knowledge is missing; it is that **none of it reaches the consumer-facing doc.**

## Friction — severity-rated and actionable

### HIGH — blocks or materially slows a producer agent

**H1. The harness workspace contract is undefined: the CLI runs in an empty throwaway temp dir, so there is no repo to edit and no diff to review.** The doc tells the PM the harness *"completed real repo bug-fix / refactor tasks end-to-end"* (`PM_LOCAL_BACKENDS.md:85`) and instructs it to *"gate every consequential harness diff through `/review`"* (`:92–93`). The runner supports neither half. It creates a fresh random temp dir (`harness-runner.js:270–271`), spawns the CLI with `cwd: dir` (`:280`), returns **only the CLI's parsed stdout text** as the task result (`:290`, `:336`; `opencode.parseResult` is literally `stdout.trim()`, `:223`), and then `rmSync(dir, {recursive:true, force:true})` in a `finally` (`:293`). There is no `git` invocation anywhere in the file. So: the temp dir starts empty — nothing copies the project into it — and whatever the CLI writes is destroyed before the result is delivered. **A PM following the doc will dispatch a coding task, receive prose, and have no diff to hand `/review`.**

There is a darker reading. The natural way for a consumer to make `:85` true is to put absolute repo paths in the payload. The child is spawned as the same uid with **no filesystem sandbox** — only `cwd` and `HOME`/`XDG_*` are redirected (`:138–149`, `:280`). So the CLI *can* reach the live working tree by absolute path, write into it with no pre-write gate, and the temp-dir teardown means the runner surfaces no record of which files changed. `/review` is post-hoc by design, which is fine — but only if the changed set is *knowable*. Right now it isn't.

*Proposed fix:* Define the workspace contract explicitly, and prefer the structural fix. **(a) Structural:** run the CLI in a **git worktree** of the project (the harness gets a real repo, concurrent tasks stay isolated — the extension already models `isolation: "worktree"` elsewhere), capture `git diff` on exit, and return it as the task result or as a `[FILE RESULT]` block pointing at the patch. That makes `:92–93` executable and gives `/review` something to audit. **(b) Minimum:** if the harness is intended to be text-only, say so plainly, retract or re-scope the "real repo bug-fix / refactor tasks end-to-end" claim to describe the maintainer's validation rig rather than consumer behavior, and state that a harness agent must not be pointed at a live tree. Either way, the FS-containment question should get one explicit sentence — silence here reads as "it's handled," and it isn't.

**H2. `local-runner.js` and `harness-runner.js` implement the same SSRF threat model at two different strengths; the C-1 path is not DNS-rebind-safe.** `harness-runner.js` resolves the upstream hostname once and **pins the IP**, forwarding to `host: pinnedIp` (`:106`, `:120`), with an allow-list on the *resolved* address (`:79–90`) and a deny that precedes it (`:73–78`, `:108`). `local-runner.js` instead checks the **hostname string** (`isPrivateHost`, `:58–70`) and then lets the HTTP client resolve independently at connect time (`:184–194`) — the classic rebind window. The key-env allow-lists have also drifted apart: `harness-runner.js:64` accepts `LM_STUDIO_|TASK_ROUTER_HARNESS_|TASK_ROUTER_BACKEND_|LOCAL_`, while `local-runner.js:49` omits the `TASK_ROUTER_HARNESS_` prefix. Two implementations of one invariant will keep diverging.

**Precondition, stated plainly so this is not read as alarmism:** the rebind window opens **only when the endpoint is a DNS hostname** — reached via `TASK_ROUTER_BACKEND_HOST_ALLOW`, or a literal `localhost`. The documented default `http://127.0.0.1:1234/v1` is an **IP literal**, so node performs no lookup and cannot rebind. We are therefore *not* asserting a live vulnerability in the default deployment. We rate this HIGH on the **divergence** argument rather than the exploitability one: two implementations of one security invariant will keep drifting apart, and the next endpoint someone configures by hostname inherits the weaker of the two. Note also that the `API_KEY_ENV_ALLOW` drift is itself **benign in effect** — `local-runner.js` is merely *more* restrictive, over a prefix it never uses. We flag it as evidence of divergence, not as a hole.

*Proposed fix:* Lift the resolve-and-pin logic, the metadata deny-list, and the `API_KEY_ENV_ALLOW` regex into a single shared module that both runners import. The harness implementation is the correct one; promote it rather than re-deriving it.

**H3. Nothing preflights the harness, so every misconfiguration surfaces as a slow per-task failure rather than a startup error.** Three checks are absent from `harness-runner.js`. **(i) Binary existence:** the bin name is resolved (`:275`) but never checked before spawn; a missing `pi`/`opencode` becomes `SPAWN_ERR` (`:286`) → a per-task error result (`:289`). A fleet whose host lacks the CLI will register the agent, accept dispatches, and fail each one individually. **(ii) Endpoint reachability:** the proxy DNS-resolves and IP-pins (`:105–109`) but never test-connects; `--selftest-proxy` (`:388`) exists but is manual and outside `main()`. **(iii) Model tier:** none — see M3. Compounding this, the doc's own warning that *"a ~12B cannot drive the loop (**it hangs**)"* (`:68`) meets a default `timeout_ms` of **900 000** (`harness-runner.js:309`, doc `:74`). So the documented misconfiguration costs **15 minutes of wall-clock per task** before returning `harness timed out after 900000ms` (`:288`) — with no hint that the model tier is the cause. Meanwhile `/pm audit` is scoped to the *task-shaping* contract only (`PM_LOCAL_BACKENDS.md:104`), so it will not catch any of this.

*Proposed fix:* On agent registration (or first dispatch), preflight: `which <bin>` → refuse to register with a clear error; one cheap `GET /v1/models` against the endpoint → warn if unreachable, and warn if the pinned `model` is absent from the returned list. Extend `/pm audit` so that **when a `harness` agent is present** it reports CLI-on-PATH and endpoint-reachable as findings. Separately, consider a much shorter *first-token* timeout distinct from the 15-minute total: a model that hangs produces no tokens at all, which is cheaply distinguishable from a model that is working slowly.

**H4. The harness computes an `ok` flag from the CLI's exit code and then never tells anyone. A failed run is delivered to the PM as an ordinary result.** `parseResult` returns `{ ok: code === 0, result }` in **both** adapters (`harness-runner.js:207` for pi, `:223` for opencode). But `main()` transmits only the result — `complete_task(…, result: r.result)` (`:336`). `r.ok` is consumed locally for a log line (`r.ok ? 'completed' : 'reported error'`, `:336`) and to gate one stderr append (`:291`, first 400 chars, **only when `!ok` and stderr is non-empty**). Nothing carries the exit status across the task boundary. Consequence: a CLI that exits non-zero, prints plausible-looking output to stdout, and writes nothing to stderr produces a task result **indistinguishable from success**. The PM has no signal to gate on, and the `/review` step the doc mandates (`PM_LOCAL_BACKENDS.md:92–93`) is handed a failed run with no marker saying so.

This sharpens the doc's own epistemics rather than contradicting them. `:93` warns that *"a clean `exit 0` from the CLI is not correctness."* True — and understated. **The PM never sees `exit 0` in the first place.** It cannot even reach the weaker conclusion the doc is warning it not to draw. Note that spawn failures and timeouts *do* surface, because those paths write a JSON error string into `result` itself (`:288`, `:289`); it is specifically the ordinary non-zero exit that vanishes.

*Proposed fix:* Propagate the exit status into the task record — either as a structured result envelope (`{ ok, exit_code, result }`) or, minimally, by prefixing a machine-greppable marker such as `[HARNESS-FAILED exit=<code>]` to `result` whenever `!ok`, so a PM can refuse to route a failed run to `/review` as if it were a diff. Appending stderr only when it happens to be non-empty is not a substitute.

**H5. Every harness dispatch is a cold start, and the knowledge-accumulation passage is ambiguous about whether that is intended.** The runtime fact is not in doubt. Four independent mechanisms make **cross-task** accumulation impossible. **(i)** pi's argv is hardcoded with `--no-session` (`harness-runner.js:175`); opencode is invoked as one-shot `run` (`:222`). **(ii)** Each task gets a fresh random temp dir (`:270–271`). **(iii)** `childEnv` redirects `HOME`, `XDG_CONFIG_HOME`, `XDG_DATA_HOME`, `APPDATA`, `LOCALAPPDATA` **into that same per-task dir** (`:138–149`), so any session store either CLI would write lands there. **(iv)** The dir is `rmSync(…, {recursive:true, force:true})`-deleted in a `finally` (`:293`). The mechanisms are over-determined: even if `--no-session` meant something narrower than we assume, the redirected-and-deleted config home would still destroy the state. Nothing survives a task but the next task's stdin payload.

**We cannot tell whether `:97–100` claims otherwise, and we want to be careful here rather than accuse the doc of a bug it may not have.** The passage says the ≈90K window *"comfortably holds a specialist's accumulated session (typically tens of thousands of tokens), so a harness specialist can carry context across its work,"* framing latency as the accepted trade for *"correctness and accumulation."* Two readings survive contact with the source:

- **Cross-task** — the agent accumulates across dispatches, the way a Claude specialist terminal does. Under this reading the passage is unachievable as shipped, per (i)–(iv).
- **Within-a-single-dispatch** — the window holds one agentic loop's working context (files read, intermediate reasoning) while rule 3 keeps the *input* shortlisted. Under this reading **the runner is entirely consistent and there is no bug.**

The passage's own phrase *"keep each **dispatch's** input shortlisted"* leans toward the second. We have run zero harness tasks and cannot resolve the author's intent, so we are **not** asserting a defect — we are asking for disambiguation. What we can say firmly is that the wording *"accumulated session"* and *"carry context across its work"* invites the first reading, and a PM who takes it will scope and brief the agent wrongly.

The consequence is real either way, and it is why this is HIGH rather than a doc nit: **a harness specialist must be re-briefed from `Context docs:` on every single dispatch.** That interacts badly with our M2 recommendation to carve a narrow `capabilities:["code"]` harness specialist. Much of what makes the seed's own Agent-Evolution economics favor specialists is warm accumulated context; a harness specialist has none. It is a stateless function call wearing an agent's name. Defensible — arguably the safer default — but it should be stated, because it changes how the agent is scoped, prompted, and costed.

*Proposed fix:* Disambiguate `:97–100` in one sentence. If statelessness is intended (we assume it is, and `--no-session` looks deliberate), say plainly that a harness agent is **stateless per dispatch**, that the ≈90K window bounds a *single* dispatch rather than an accumulated session, and that the PM must front-load context into each payload — which collides with rule 3 (*bound the INPUT*) and deserves an explicit word on resolving that tension. If cross-task continuity was intended, then the per-task temp dir, the config-home redirection, and `--no-session` are all working against it.

**H6. Nothing in the seed tells a consumer how to create a local agent, and `PM_LOCAL_BACKENDS.md` — the only sane home for that knowledge — is loaded on a trigger that makes it structurally incapable of being one. Following it exactly also makes a PM violate its own mandatory rule 2 without ever learning it did.**

**Our expectation, stated plainly, because it is the frame for everything below.** `PM_LOCAL_BACKENDS.md` should be **the rulebook for creating and operating C-1 and C-2 local agents** — the one document a PM reads to stand one up, configure it, route to it, and know what it may and may not be asked to do. It is not that today. We are not asking for a pointer to a second document; we are asking for this one to be sufficient. (We considered proposing a split — a separate `LOCAL_AGENT_SETUP.md` — and rejected it: a second doc needs its own load trigger, will drift from the routing rules it must stay consistent with, and reintroduces the discoverability problem in L1. One authority, lazily loaded, is the right shape.)

**The load trigger is the root cause, and it is a genuine chicken-and-egg.** The doc's own `Load when` line reads *"a local `backend`/`harness` agent **is present** and you are about to route work to it"* (`PM_LOCAL_BACKENDS.md:10`), and the sole pointer to it gates identically — *"**If any agent has** a `backend` (or `harness`) block, read `PM_LOCAL_BACKENDS.md` before routing work to it"* (`PM.md:533`). Both fire only **after** the agent exists — and `PM.md:533` is the **only PM-facing pointer to `PM_LOCAL_BACKENDS.md` in the entire seed** (verified: `grep -rn PM_LOCAL_BACKENDS PM.md PM_TEMPLATES.md .claude/mcp/task-router/doc/ .claude/skills/` returns exactly that one hit). The Agent-Evolution and agent-add paths in `PM_TEMPLATES.md` — where a PM actually decides to create an agent — never mention local-model agents. So a PM deciding *whether* to add one, or attempting to create one, is never told to read the only document about them. We draw no conclusion about how the doc came to be this way. We observe only that **fixing the trigger is a precondition for fixing the content**: anything added under a predicate that cannot fire at creation time is unreachable by the reader who needs it.

We tested sufficiency directly: our operator asked whether the doc contained enough to stand up a C-1 and a C-2 agent. It does not, for either class, and the failures below are ordered by how quickly they stop you.

**(i) A local agent is a resident process that nothing in the seed starts or supervises, and the doc never says so.** Both runners are self-driving pollers — `for (;;)` around `pickup_next_task` on a 3 s interval (`local-runner.js:230`, `harness-runner.js:323`) — and each **self-registers on startup** via the `register_agent` MCP tool (`local-runner.js:223`, `harness-runner.js:318`), idempotently, precisely so the launcher-less case works. Read against the doc, though, a local agent looks like a *configuration block*. It is in fact a daemon that must be started and kept alive, and **nothing in the shipped seed spawns or restarts either runner** — a repo-wide grep for `local-runner`/`harness-runner` finds only prose.

Your own header comment states the contract the doc omits: *"A launcher pre-registers the agent and sets the env below, then spawns `node local-runner.js`. **In C-0 an operator places the file and env by hand**"* (`local-runner.js:11–14`), and, flagged **AUTONOMY (audit CRITICAL)**, *"a one-shot that exits is never relaunched (the nudge loops only inject into live sessions; **nothing spawns a dead runner**)"* (`:16–19`). A `--reseed` consumer with no VSIX is therefore in **C-0** — a named mode, with real operational obligations — and `PM_LOCAL_BACKENDS.md` mentions neither C-0, nor the launcher, nor the fact that a process must exist at all. The operational spec is complete and well-written; it simply lives in a source-file header that the consumer-facing doc never points at.

**(ii) The runners are driven entirely by environment variables the doc never names.** Twenty distinct `TASK_ROUTER_{BACKEND,HARNESS}_*` variables (eleven backend, nine harness — `ENDPOINT`, `MODEL`, `KIND`, `CAPABILITIES`, `API_KEY_ENV`, `TIMEOUT_MS`, `POLL_MS`, `HOST_ALLOW`, plus `DRIVER`/`TEMPERATURE`/`MAX_TOKENS` on C-1 and `BIN` on C-2), over four shared ones. `PM_LOCAL_BACKENDS.md` mentions **zero** of them; it speaks exclusively in `agents.json` field names.

The `capabilities` case shows why that matters. Registered capabilities come from `TASK_ROUTER_*_CAPABILITIES` (`local-runner.js:223`, `harness-runner.js:318`); **neither runner ever reads `agents.json`.** The array in the doc is presumably the intended *source*, with a launcher translating array → env — but per L2 no launcher ships, so in C-0 the array drives nothing. A C-0 backend agent self-registers with **empty** capabilities (`local-runner.js:223` defaults to `''`; the harness defaults to `'code'`). Rule 1 — *"only route a task whose class is in the agent's `capabilities`"* — is thus written against an array that, in the mode a `--reseed` consumer is actually in, is never consulted by anything.

**(iii) Rule 2 is undiscoverable, and its default silently fails.** This is the serious one. Rule 2 is stated as mandatory: *"Every task must be `json_schema`-outputable with a **bounded** schema — enums, typed scalars, `maxItems` on arrays, `maxLength` on strings"* (`PM_LOCAL_BACKENDS.md:36–37`). **The doc never shows how a schema reaches the model.** The actual contract lives only in source (`local-runner.js:157–183`): a payload is either plain text, or a JSON object `{ system?, prompt|user, schema? }` supplying its own schema. A plain-text payload — the obvious thing a PM sends — silently receives `DEFAULT_SCHEMA` (`:157`), which is `{ result: { type: 'string' } }`: **one unbounded string, no `maxLength`.** That is precisely the degenerate case rule 2 exists to prevent, and precisely the one the doc warns *"the model degenerates into repetition/hallucination even on an easy task"* (`:37–38`).

So a PM that reads this document carefully, routes a whitelisted task, and writes a sensible prompt **violates the document's own mandatory rule**, gets no warning, and has no way to discover the payload convention that would have satisfied it. Rule 2 is not difficult to honor. It is impossible to honor from the doc.

**(iv) Smaller things a consumer must currently guess.** `driver: "minimal"` is presented as a field with options; it is the only accepted value and anything else is fatal (`local-runner.js:205–206`). `max_tokens` defaults to **800** (`:179`) and will silently truncate a longer bounded output. `temperature` defaults to 0.2 (`:178`). And nothing states whether a local agent needs a `SKILL.md`, a rules file, a `GUIDELINES.md`, or a Doc-Ownership-Matrix row — the agent-add recipe in `PM_TEMPLATES.md` has no local-agent variant, and its canonical specialist blocks (Startup Sequence, Memory Policy, Code Review) are meaningless for a process that never holds a Claude context.

*Proposed fix:* **Make `PM_LOCAL_BACKENDS.md` the rulebook it ought to be** — self-sufficient for creating, configuring, and operating a C-1 or C-2 agent, with no recourse to source. Five changes, the first of which unblocks the rest:

1. **Fix the load trigger.** Extend `Load when` (`:10`) and the `PM.md:533` pointer to fire on *considering or creating* a local agent, not only on routing to an existing one — e.g. *"…is present, **or you are evaluating whether to add one**."* Wire it into the Agent-Evolution path in `PM_TEMPLATES.md`, which is where a PM decides to create an agent at all. Without this, no amount of added content is reachable at the moment it matters.
2. **Say that a local agent is a resident process.** That it self-registers but that *nothing spawns or restarts it*; what C-0 obliges an operator to do; what supervises it; how it is stopped. This is the single largest missing fact, and your `local-runner.js:11–19` header already states it precisely.
3. **Publish the `TASK_ROUTER_{BACKEND,HARNESS}_*` env-var table.** The runners' header comments already document all twenty variables clearly (`local-runner.js:21–32`; `harness-runner.js:30–48`). **The spec exists and is good — it simply never reaches the consumer-facing doc.** Lifting those two comment blocks into `PM_LOCAL_BACKENDS.md` would close most of this item. State which source is authoritative when `agents.json` and the environment disagree, and settle `capabilities` (see (ii)).
4. **State the C-1 payload contract adjacent to rule 2, with a worked bounded example.** A rule is only mandatory if it is actionable where it is asserted. Show a real `{system, prompt, schema}` payload with an enum, a `maxLength`, and a `maxItems`, so a PM can copy it.
5. **Say whether the agent-add recipe applies.** A local agent plainly needs no Startup Sequence, Memory Policy, or Code Review block. Does it need a `SKILL.md`, a rules file, a `GUIDELINES.md`, a matrix row? One sentence closes this.

Separately, and independent of documentation: **bound the C-1 default.** We are not asking you to refuse schema-less tasks — the plain-text convenience path is clearly deliberate (`local-runner.js:159–160`) and deleting it would be the wrong trade. The narrow ask is that the *default itself* comply with the rule the doc marks mandatory: give `DEFAULT_SCHEMA`'s `result` a `maxLength`, so the convenience path is rule-2-compliant by construction rather than in violation of it. A loud warning when no schema is supplied would be a reasonable addition; refusing the task would not. A default that silently contradicts a rule labelled *mandatory* is the wrong default, however convenient — but the fix is to bound the default, not to remove it.

### MEDIUM — would improve output quality or reduce consumer friction

**M1. Rule 2 is labelled mandatory and phrased universally, but cannot hold for C-2 harness tasks.** *"Every task must be `json_schema`-outputable with a **bounded** schema"* (`PM_LOCAL_BACKENDS.md:36–37`), under the heading *"The four shaping rules (**mandatory**)"* (`:32`). A C-2 harness task emits free-form CLI text (`harness-runner.js:290`; `opencode.parseResult` = `stdout.trim()`, `:223`) — there is no schema to bound. v4.32 inserted the C-2 envelope *above* rule 2 without reconciling rule 2 to it. This is not cosmetic: rule 2 is precisely what makes rule 4's `/review` gate coherent for C-1 ("format guaranteed, content unverified"). For C-2 the entire correctness burden falls on `/review` plus a test suite, with **no schema backstop at all** — and the doc does not say so, leaving a PM to believe a guarantee is present that isn't.

*Proposed fix:* Scope rules 2 and 3 explicitly to C-1, and add a C-2 rule in the same numbered list: *"**Bound the ACCEPTANCE.** A harness task has no schema backstop. Never route one without an execution-graded acceptance signal — a test that must pass — and never accept `exit 0` as that signal."* This also gives the `/review` gate something concrete to check.

**M2. The envelope is defined per-task; the model pin is per-agent. The doc never reconciles the two, so the operator's natural question has no answer in it.** Our operator asked, verbatim, which specialists could be diverted to a local 31B. Rule 1 binds routing to an agent's `capabilities` (`:34–35`), while `agents.json` binds the model to the *agent*. Therefore repinning an existing specialist routes its **entire** scope to the local model — including the cross-cutting refactors the doc explicitly excludes (`:90`). No existing specialist's full scope fits the envelope, so the answer is always "none," and the correct move is to **carve a narrow new capability agent** out of the domain. The doc's own example does exactly this (`"coder": { capabilities: ["code"], harness: {…} }`, `:72–74`) — but it never states the principle, so the example reads as arbitrary.

*Proposed fix:* One paragraph after the `coder` example: *"You do not divert an existing specialist to a local model. An agent's model pin and its capability list move together, so an agent is divertible only if its **entire** scope fits the envelope — which a domain owner's never does. Instead, carve a narrow `capabilities: ["code"]` harness agent out of the domain, and leave the domain owner on Claude."*

**M3. "31B-tier" and "dense vs sparse-MoE" are load-bearing predicates that a consumer cannot evaluate and the runner does not check.** The doc makes architecture decisive (`:91`), but `model` is an **opaque passthrough**: `harness-runner.js:269` charset-validates it solely to prevent command injection under `shell:true`, then writes it straight into the CLI config (`:171`/`:215`) and `--model` (`:175`/`:222`). Nothing inspects size, density, or tier. A consumer reading an LM Studio model list cannot tell dense from MoE from the id — ours offers `qwen3.6-14b-a3b`, whose id advertises `14b` while activating ~3B, which by the doc's own reasoning is the *worst* case (fails silently). The doc warns about the category but names no member of it.

*Proposed fix:* Either publish a short eligibility list ("measured pass: …; measured silent-fail: …") in the doc, or add a declarative `harness.tier` / `harness.arch` assertion field that the runner logs and warns on when the pinned model id doesn't plausibly match. Even an unenforced field turns an invisible assumption into a reviewable one. At minimum, name in the doc which architectures were measured to fail the composite task — the negative result is the actionable half.

**M4. The resident harness supervisor buffers child stdout/stderr unbounded — a runaway or hostile model can OOM it and take the whole agent offline.** `child.stdout.on('data', (d) => (out += d))` and the matching `stderr` handler (`harness-runner.js:281`) accumulate into strings with **no byte cap**. The 15-minute hard timeout (`:284`) bounds *time*, not *memory*. The task payload is untrusted and arrives on stdin (`:282`); a prompt that drives the coding CLI into an unbounded output loop — exactly the *"repeats/hallucinates"* degeneration the doc warns about for unbounded fields (`PM_LOCAL_BACKENDS.md:29`) — can exhaust the supervisor's heap **before** the timeout fires. The blast radius is the wrong size: this kills the **resident** supervisor process, not just the one task, and nothing auto-relaunches it. A single bad prompt takes the agent off the roster.

*Proposed fix:* Cap accumulation — truncate `out`/`err` past a few MB and mark the result truncated — or stream to a temp file with a size ceiling. This pairs naturally with the shorter first-token timeout proposed in H3: a model that hangs emits nothing, a model that degenerates emits everything, and the two failures want different bounds.

**M5. The C-1 payload channel is also a privilege surface: a payload beginning with `{` replaces the system prompt, so an untrusted origin can repurpose the model.** Distinct from H6(iii), which is about a bound a PM *cannot find*; this is about a bound a PM *sets and an attacker removes*. `buildCompletion` parses a JSON payload and adopts caller-supplied `system`, `prompt`/`user`, **and** `schema` (`local-runner.js:161–183`, its contract described at `:159–160`). The `schema` override is necessary — per H6(iii) it is the only way to satisfy rule 2 at all. The `system` override is what concerns us. Together they mean "Bound the OUTPUT" (`PM_LOCAL_BACKENDS.md:36–37`) is a **convention held by the dispatcher, never an invariant held by the runtime**.

For a fleet where every payload originates with a careful PM, this is fine. Ours would not be that fleet: our inbound requests arrive tagged `[HA REQUEST]` from Home Assistant and as free text from a Telegram bridge. **To be clear about our own exposure — we run no `backend` agent today, and no launcher ships that could spawn one, so this is a forward-looking hazard for us rather than a live hole.** But it is precisely the deployment shape we would use one in, and we suspect we are not unusual: any dispatch path that lets attacker-influenced text reach a `backend` agent's payload verbatim hands that attacker both the system prompt and the output schema of a local model — dissolving the control rule 2 rests on, in the one deployment shape (private, offline, home-automation) the C-1 backend is explicitly *for*. The doc never mentions the surface exists.

*Proposed fix:* Two independent changes, both cheap. **(a)** Document the override surface beside rule 2, and state plainly that a payload forwarded from an untrusted origin must be re-wrapped by the dispatcher, never passed through. **(b)** Let the `backend` block pin a schema ceiling and, separately, a fixed `system` prompt that a payload may **narrow but never replace** — that turns rule 2 from advisory into structural, and closes the system-prompt hijack outright.

**M6. `pi` and `opencode` are interchangeable at the security layer but NOT at the result-contract layer, so "the PM just routes" is not accurate.** `PM_LOCAL_BACKENDS.md:94–96` tells the operator to *"pick whichever CLI is installed on the fleet host"* and assures them *"the runner handles the CLI-specific invocation and the security proxy; **the PM just routes**."* The first half is true and well-built — payload-on-stdin (`harness-runner.js:282`), timeout (`:309`), proxy, key injection, and env scrub are all adapter-agnostic, handled once in `runHarness`. The second half is not. The two adapters return **different kinds of string** for the same task:

- **pi** (`:176–208`) performs real structured extraction over pi's `--mode json` newline-delimited event stream: it locates the `agent_end` event, scans `messages[]` backward for the last `role === 'assistant'` entry, filters its `content` to `type === 'text'` parts, and joins them — deliberately **dropping `thinking`/reasoning** (`:196`). Unparseable lines are silently skipped (`:184`), with two documented fallbacks ending in raw `stdout.trim()` (`:206`).
- **opencode** (`:223`) is `(stdout, code) => ({ ok: code === 0, result: stdout.trim() })`. **No parsing at all.** We want to be precise about what this does and does not imply. Your own comment at `:37–38` records that you verified opencode end-to-end on a 31B and that *"Output parses clean via `stdout.trim()`"* — you have run it and we have not, so that observation is stronger evidence than any inference of ours. Our point is narrower: **the runner does not enforce output cleanliness for opencode, it relies on it.** Should a future opencode version emit a banner, a progress line, or a deprecation warning on stdout, it would pass through into the task result unfiltered, and nothing in the runner would catch it. pi is structurally immune to that class of regression because it extracts from a typed event stream.

So a PM that swaps `kind` receives a *different kind of string* for the same task: from pi, reduced assistant text with thinking stripped; from opencode, whatever landed on stdout. Compounded with **H4** (the `ok` flag is never transmitted), opencode is also the weaker of the two on failure signaling — pi at least loses its JSON structure when something goes wrong, whereas raw stdout on a non-zero exit with empty stderr looks exactly like a normal answer. The config surfaces differ non-trivially too: pi writes `.pi/agent/models.json` with `models: [{ id }]` and `baseUrl`/`api`/`apiKey`, discovered implicitly via cwd plus the redirected config-home (`:169–172`); opencode writes a flat `opencode.json` with `models: { [model]: {} }`, `options.baseURL`, an `npm` provider field, and requires an explicit `--dir` because it will not discover config from CWD under a cmd.exe spawn (`:212–222`, comment `:218–221`).

*Proposed fix:* Replace *"the PM just routes"* with an honest statement of what differs, and name the result contract per adapter (pi: reduced assistant text, thinking stripped; opencode: raw stdout). If they are meant to be substitutable, give opencode a real `parseResult` so both honor one contract. **Scoped recommendation, and we want the scope explicit:** on *adapter-integration robustness* alone — structured extraction, thinking stripped, graceful fallbacks (`:198–206`), and better failure signaling under H4 — pi is the more defensively built of the two, and we'd suggest naming it the recommendation rather than merely the default. We are **not** claiming pi produces better code than opencode: the model drives the coding, the adapter only carries the result. We have run neither.

### LOW — polish suggestions

**L1. The lazy-load discipline hides the harness concept a little too well.** `grep -rn "harness" PM.md PM_TEMPLATES.md .claude/mcp/task-router/doc/` returns exactly **one** substantive hit: the conditional pointer at `PM.md:533`. Every other match is an unrelated use of the word — the *"harness-managed auto-memory directory"* (five occurrences) and *"single-Read limits in many harnesses"* (`PM_TEMPLATES.md:11`). `pi` and `opencode` appear in **zero** seed files outside `PM_LOCAL_BACKENDS.md` itself. A PM that has never seen a `harness` block has no way to learn that local agentic coding is an option at all — the doc is discoverable only once you already have the thing it describes.

*Proposed fix:* One sentence in `PM_TEMPLATES.md`'s routing/Agent-Evolution section noting that a local `backend` (bounded, C-1) or `harness` (agentic coding, C-2) agent is an option, pointing at `PM_LOCAL_BACKENDS.md`. Costs ~30 tokens against the lazy-load budget and makes the feature findable. Note this is the same sentence H6's fix (1) needs, for a different reason: L1 is about a PM never *discovering* the feature, H6 is about a PM never being told to *read the rules* before creating one. One edit closes both.

**L2. Nothing in the shipped seed translates the `agents.json` `harness` block into the env vars the runner actually reads.** The doc's example carries `"timeout_ms": 900000` and `"kind": "pi"` (`:73–74`), but the runner reads `TASK_ROUTER_HARNESS_TIMEOUT_MS` (`harness-runner.js:309`) and `TASK_ROUTER_HARNESS_KIND` (`:303`) **from the environment**. A repo-wide grep for `TASK_ROUTER_HARNESS_` finds exactly one file — `harness-runner.js` itself. There is **no launcher, hook, or mapping layer** anywhere in what we received. So a consumer editing the `harness` block in `agents.json` cannot tell whether those fields are live or decorative, and on this evidence they are decorative. (This also decides the first doc bug below: nothing supplies the `"pi"` default the doc promises.) We assume the mapping ships with the VSIX and simply isn't visible to a `--reseed` consumer — worth confirming.

*Proposed fix:* State where the `harness` block → `TASK_ROUTER_HARNESS_*` translation happens and name the variables, so a consumer can tell configuration from decoration. If the mapping lives only in the extension, say that explicitly — a `--reseed`-only fleet has no working harness path at all.

## Possible documentation bugs

- **`PM_LOCAL_BACKENDS.md:67`** — *"**`kind`** = `"pi"` (default) or `"opencode"`."* The runner has **no default**: `TASK_ROUTER_HARNESS_KIND` is required, and its absence is fatal — `die('TASK_ROUTER_HARNESS_KIND not set (pi|opencode)')` (`harness-runner.js:303`), with an unknown value dying at `:304`. We went looking for a launcher that might supply the default and **found none in the shipped seed** (see L2 — the grep hits only the runner). On the evidence we have, the "(default)" is unsupported by any code we received.
- **`PM_LOCAL_BACKENDS.md:79` vs `:73`** — *"The harness CLI (`pi` / `opencode`) must be installed on **the fleet host**"* sits directly beneath an example whose `endpoint` is `http://192.168.1.50:1234/v1`, i.e. **a different machine**. Two hosts are in play — the box running the CLI and the box serving the model — and the doc names only one. This cost us a materially wrong assessment: our PM initially concluded a 31B was impossible because the *daemon* host has 16 GB RAM, before realising the model may be served from anywhere reachable and the daemon host only needs the CLI binary. Suggest naming the two roles explicitly ("harness host" vs "inference host").
- **`PM_LOCAL_BACKENDS.md:85`** — *"completed real repo bug-fix / refactor tasks end-to-end through both Pi and OpenCode."* As shipped, the runner gives the CLI an empty temp dir and deletes it (see **H1**). We cannot reconcile this sentence with `harness-runner.js:270–293`. Flagging as a possible doc bug rather than a runner bug because the measurement plainly happened somewhere — most likely in a rig that provided a workspace the shipped runner does not. Maintainer confirms which side is authoritative.

## Suggested additions

**A1. Concurrency and serialization guidance for a single inference host.** The doc is silent on whether two harness tasks may run simultaneously against one LM Studio instance. The runner isolates them at the *filesystem* layer — distinct random temp dirs, per-task redirected `HOME`/`XDG_*` (`harness-runner.js:270`, `:138–149`) — but nothing serializes them at the *inference* layer. On a single-GPU host the second task's prefill queues behind the first, and the 15-minute timeout (`:309`) is per-task and starts at spawn, not at first token. Two concurrent 31B harness tasks on one card could therefore both time out while both are making progress. We'd expect this to live beside the *Turnaround expectation* section (`:46–50`), which today models a single call in isolation. Sketch: a note on whether the supervisor caps in-flight harness tasks per endpoint, plus advice to set `timeout_ms` against *queued* rather than *solo* latency.

## What not to change — explicit preservation

- **`backend` / `harness` / `launcher` as mutually exclusive `agents.json` blocks** (`:54`). This turned a safety judgment into a schema invariant for us. Our privileged HA config-write deployer (`no_fork`, cap-token, its own `launcher`) **structurally cannot** carry a `harness` block — the schema forbids it before anyone has to argue about whether spawning a child CLI from a token-holding principal is wise. A future round that relaxes this to "an agent may have both" would silently delete that guarantee. Keep them exclusive.
- **The real API key never reaching the child process** (`harness-runner.js:111`, `:117–118`, `:171`, `:215`). The CLI is handed a per-spawn random client token as its `apiKey`; the proxy strips the incoming auth and injects the real key only on forward. This is the right shape and it should survive any refactor that "simplifies" the proxy away.
- **The child-env allowlist plus `--selftest-scrub`** (`:138–149`, `:350–359`). An allowlist that ships with an executable proof it holds is rare. Keep the selftest; it is what makes the claim auditable rather than aspirational.
- **Untrusted payload via stdin, never argv** (`:282`), given `shell: true` is in play.
- **`"json_schema` guarantees *format*, never *content*"** (`:12–13`) and **"a clean `exit 0` from the CLI is not correctness"** (`:93`). Both are the kind of sentence a later editor trims as redundant. They are not — they are the load-bearing epistemics of the whole document.
- **The `❌` column framing itself.** A capability doc that leads with what the thing *cannot* do is unusual and correct. Please resist reframing it as a capabilities-first table.

## Meta-note

The errata channel worked exactly as designed on this round and is worth saying so: v4.32 landed as a clean, idempotent, doc-only delta with a stated verification predicate, and the mandatory apply/decline/defer prompt meant the operator saw the version delta before anything was written. Four files, `+30/−6`, reviewable as one diff.

The forward-looking risk we want on your radar is **the gap between a shipped runtime and a shipped capability**. `harness-runner.js` arrived here today, well-engineered and security-hardened, into a fleet with no 31B, no CLI, and no harness agent — and into a doc that already describes end-to-end repo edits (`:85`) the runner as written cannot perform (**H1**). That ordering is fine for a runtime that ships ahead of demand. It is dangerous for a *doc* that ships ahead of the runtime, because the doc is what a PM routes on. A PM reading `:85` and `:92–93` will confidently dispatch a coding task and confidently wait for a diff. Ours nearly did.

The narrower version of the same concern: v4.32's headline is a **capability expansion** ("31B clears agentic coding"), and capability expansions are read as permission. The three sentences that make it safe — architecture-over-size, silent failure, `exit 0` is not correctness — are doing enormous work in a bulleted aside. We would rather see them promoted than see the ❌ column shrink again in v4.33.

One structural observation that ties **M4** and **M5** together, since they were found late and read as unrelated: both are places where a **bound the doc asserts is not a bound the runtime holds**. Rule 2 tells the PM the output is bounded, but `local-runner.js` lets the payload replace the schema (M5). The timeout tells the PM a runaway task is contained, but the containment is temporal and the buffer is not (M4). In both cases the doc describes an invariant and the code implements a convention. That is a survivable gap while the dispatcher is a careful Claude PM — it stops being survivable the moment an untrusted origin reaches the payload, which for a Home-Assistant or chat-bridged fleet is the normal case, not the exotic one. If one theme from this round is worth carrying into v4.33, it is that one.

Finally, a process note: the pre-submission checklist's instruction to `grep`-check every "gap" before filing it caught two would-be items in this round (both were navigability, not absence) and forced the recon pass that produced **H1** and **H2**. It is the highest-leverage item in the template. Keep it.
